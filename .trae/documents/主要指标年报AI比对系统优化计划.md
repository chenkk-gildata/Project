# 主要指标年报AI比对系统优化计划

## 一、需求概述

### 当前问题

原系统读取单个PDF交给AI抽取数据后与数据库比对。现因AI抽取效果问题，将处理拆分为两个模块：

* **模块一**：处理指定目录下的PDF文件，使用test1提示词提取主要会计数据和财务指标

* **模块二**：处理指定目录下"净资产收益率和每股收益"子目录中以"\_mgsy"结尾的PDF文件，使用test2提示词提取每股收益和净资产收益率相关数据

### 优化目标

1. 模块一和模块二并发处理，提高处理效率
2. 保证两个模块处理同一公司的文件，避免数据错配
3. 合并两个模块的提取结果，按规则处理字段冲突

***

## 二、文件结构说明

### 目录结构

```
指定目录/
├── 600000-2025-03-31-xxx.pdf          # 模块一文件
├── 600001-2025-03-31-xxx.pdf
├── 净资产收益率和每股收益/
│   ├── 600000-2025-03-31-xxx_mgsy.pdf  # 模块二文件
│   └── 600001-2025-03-31-xxx_mgsy.pdf
```

### 文件匹配规则

* 模块一文件名：`{股票代码}-{发布日期}-{其他信息}.pdf`

* 模块二文件名：`{股票代码}-{发布日期}-{其他信息}_mgsy.pdf`

* 匹配方式：去除模块二文件名的"\_mgsy"后缀后，与模块一文件名前缀匹配

## 三、模块二提取字段

根据test2提示词，模块二提取以下字段：

| 字段名           | 业务描述          |
| ------------- | ------------- |
| JZRQ          | 截止日期          |
| JBMGSY        | 基本每股收益        |
| XSMGSY        | 稀释每股收益        |
| JBMGSYKC      | 扣除后基本每股收益     |
| XSMGSYKC      | 扣除后稀释每股收益     |
| PTGJZCSYLJQ   | 普通股净资产收益率加权   |
| KCPTGJZCSYLJQ | 普通股净资产收益率扣除加权 |

***

## 四、数据合并规则

### 合并原则

模块一的字段包含模块二的字段，模块二的数据是对模块一的补充。

### 合并场景处理

| 场景  | 模块一 | 模块二 | 处理方式      |
| --- | --- | --- | --------- |
| 场景1 | 有值  | 空/无 | 使用模块一的值   |
| 场景2 | 空/无 | 有值  | 使用模块二的值   |
| 场景3 | 空/无 | 空/无 | 保持为空      |
| 场景4 | 有值  | 有值  | 按字段类型规则处理 |

### 场景4的字段处理规则

#### 每股收益类字段（*MGSY*）

包括：JBMGSY、XSMGSY、JBMGSYKC、XSMGSYKC

处理逻辑：

```
IF 模块一精度 == 模块二精度 THEN
    以模块二数据为准
ELSE
    比较数据精度（小数点后位数越多精度越高）
    以精度高的数据为准
END IF
```

精度判断方法：计算小数点后的位数

#### 净资产收益率类字段（*JZCSYL*）

包括：PTGJZCSYLJQ、KCPTGJZCSYLJQ

处理逻辑：

```
以模块二数据为准
```

### 5.2 处理流程重构

#### 原流程

```
遍历PDF文件 -> 上传 -> AI提取 -> 数据库查询 -> 比对 -> 生成报告
```

#### 新流程

```
1. 文件配对阶段
   - 扫描指定目录下的PDF文件（模块一）
   - 扫描"净资产收益率和每股收益"子目录下的*_mgsy.pdf文件（模块二）
   - 按公司标识（股票代码-发布日期）配对文件

2. 并发处理阶段（针对每家公司）
   - 并发上传模块一和模块二文件
   - 并发调用AI提取（模块一用test1提示词，模块二用test2提示词）
   - 等待两个模块都完成

3. 数据合并阶段
   - 按JZRQ匹配模块一和模块二的提取结果
   - 按合并规则处理各字段

4. 后续处理阶段
   - 数据库查询
   - 数据比对
   - 生成报告

5. 下一家公司处理
```

### 5.3 并发控制策略

#### 原有线程池配置分析

```python
# 原有配置
upload_workers = 2   # 上传线程池：最多同时2个上传任务
process_workers = 16 # 处理线程池：最多同时16个处理任务
```

**原有流程特点：**

* 使用流水线模式，上传和处理并行进行

* 上传完成后立即提交处理任务

* 使用两个独立的线程池分别管理上传和处理任务

* 高效利用资源，避免等待

#### 优化后的并发策略

**保持原有流水线架构，调整处理单元：**

* **上传线程池**：保持2个上传线程

* **处理线程池**：保持16个处理线程

* **处理单元变化**：每个处理单元从"单个文件"变为"公司文件对"

**处理流程：**

```
1. 文件配对阶段（主线程）
   - 扫描并配对模块一和模块二文件
   - 生成公司文件对列表

2. 流水线处理阶段
   - 上传线程池：并发上传各公司的文件（模块一和模块二同时上传）
   - 处理线程池：处理每家公司的文件对
     - 内部并发：模块一和模块二并发调用AI提取
     - 等待两个模块都完成后合并数据
     - 进行数据库查询和比对

3. 结果收集阶段
   - 收集所有公司的处理结果
   - 生成报告
```

**线程池配置：**

```python
# 保持原有配置
upload_workers = 2   # 上传线程数
process_workers = 16 # 处理线程数

# 新增：模块级并发配置（在处理线程内部使用）
module_workers = 2   # 模块一和模块二并发处理线程数
```

## 七、异常处理

### 7.1 文件缺失处理

* **模块一文件存在，模块二文件不存在**：正常处理，仅使用模块一数据进行比对

* **模块一文件不存在，模块二文件存在**：跳过该公司（模块一是必须的）

* **两者都不存在**：跳过该公司

### 7.2 提取失败处理

* **模块一提取成功，模块二提取失败或缺失**：正常处理，使用模块一数据进行比对

* **模块一提取失败，模块二提取成功**：跳过该公司（模块一是必须的）

* **两者都失败**：跳过该公司，记录错误日志

### 7.3 数据匹配失败

* **模块一和模块二的JZRQ不匹配**：使用模块一数据进行比对，不进行合并

* **模块一有数据，模块二对应JZRQ无数据**：使用模块一数据

* **模块一对应JZRQ无数据，模块二有数据**：使用模块一数据（以模块一为主）

### 7.4 异常处理流程图

```
开始处理公司文件
    │
    ├─ 检查模块一文件是否存在
    │   ├─ 否 → 跳过该公司，记录日志
    │   └─ 是 → 继续
    │
    ├─ 上传模块一文件
    │   ├─ 失败 → 跳过该公司，记录日志
    │   └─ 成功 → 继续
    │
    ├─ 检查模块二文件是否存在
    │   ├─ 否 → 仅使用模块一处理
    │   └─ 是 → 上传模块二文件
    │       ├─ 失败 → 仅使用模块一处理
    │       └─ 成功 → 并发提取
    │
    ├─ AI提取（模块一必须，模块二可选）
    │   ├─ 模块一失败 → 跳过该公司
    │   ├─ 模块一成功，模块二失败 → 使用模块一数据
    │   └─ 两者成功 → 尝试合并
    │
    ├─ 数据合并
    │   ├─ JZRQ匹配 → 按规则合并
    │   └─ JZRQ不匹配 → 使用模块一数据
    │
    └─ 数据库查询和比对
```

## 十、风险与注意事项

### 10.1 性能风险

* 并发处理可能增加AI API压力，需控制并发数

* 文件配对需要遍历目录，大目录可能耗时

### 10.2 数据一致性风险

* 确保模块一和模块二的JZRQ格式一致

* 确保合并逻辑不会丢失有效数据

### 10.3 兼容性风险

* 保持与现有日志、报告生成逻辑的兼容

* 保持错误处理机制的一致性
